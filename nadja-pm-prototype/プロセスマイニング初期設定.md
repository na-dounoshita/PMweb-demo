2. システムアーキテクチャ

2.1 全体構成図

本システムは4層アーキテクチャで構成される。全層をオープンソース技術または自社開発コンポーネントで構築し、特定ベンダーへのロックインを回避する。

Layer 1: データ収集層（Data Ingestion）

クライアントの業務システムからイベントログを収集し、統一フォーマットに変換してストレージに格納する。Pythonで開発するETLスクリプト群をApache Airflow（OSS）でスケジュール実行する。対応データソースは以下の通り。


* PostgreSQL / MySQL / SQL Serverトランザクションログ（SQLAlchemy経由）
* REST API経由の業務システムログ（requests + カスタムコネクタ）
* SAPイベントログ（PyRFC / SAP OData API経由）
* SharePoint / Microsoft 365ログ（Microsoft Graph API経由）
* CSVファイル手動アップロード（FastAPI アップロードエンドポイント）
* タスクマイニング録画データ（OpenRPA / カスタムPythonスクリプト）
* Make.comのWebhookログ（Webhook受信エンドポイント）


Layer 2: ストレージ層（Storage）

PostgreSQLをプライマリストレージとして使用する。ローカル環境ではDockerコンテナ、クラウドではAzure Database for PostgreSQLまたはAWS RDSを使用する。後述するスキーマ設計に基づき、イベントファクトテーブル、ケースディメンション、リソースディメンション、AIエージェント実行ログ、事前計算済みKPIテーブルを配置する。

プロセスモデルファイル（BPMN、Petri Net）はMinIO（OSS、S3互換オブジェクトストレージ）またはAzure Blob Storageに保存する。

Layer 3: 分析エンジン層（Analytics Engine）

pm4pyをコアエンジンとして、FastAPI（Python）でREST APIサーバーを構築する。Docker化してAzure Container Apps、AWS ECS、またはVPS上で実行する。プロセスディスカバリ、適合性チェック、パフォーマンス分析、バリアント分析、根本原因分析をAPIとして公開する。

Layer 4: 可視化層（Visualization）

Metabase（OSS）をメインのダッシュボードエンジンとして使用する。PostgreSQLへの直接接続でほぼリアルタイムのデータ反映を実現する。プロセスマップ等の高度なビジュアルはStreamlit（Python）で補完する。クライアント向けにはMetabaseのEmbed機能またはStreamlit Cloudでダッシュボードを配信する。

3. データベーススキーマ設計

3.1 設計思想

本スキーマは、学術的なDB-XESモデル、OCEL 2.0オブジェクト中心モデル、PMCubeスノーフレークスキーマの各パターンを統合した「ハイブリッド スター/OCELアーキテクチャ」を採用する。中心にイベントファクトテーブルを配置し、リソース・ケース・エージェントのディメンショナルテーブルで囲む構造である。既存のプロセスマイニングツールにはない「AIエージェント実行ログ」と「自動化KPI」テーブルが本システムの独自要素となる。

4.3 データフロー

1. FastAPIがPostgreSQLからイベントログをpandas DataFrameとして読み込む（SQLAlchemy + pandas.read_sql）
2. pm4pyのformat_dataframe()でXES標準カラム名にマッピング
3. 指定された分析メソッドを実行
4. 分析結果をPostgreSQLのKPIテーブルに書き戻し、プロセスモデルファイルはMinIOに保存
5. MetabaseがPostgreSQLに直結して最新データを表示

 e.event_timestamp,
 e.end_timestamp,
 e.resource_id,
 e.resource_type,
 e.activity_cost,
 e.lifecycle,
 e.is_automated,
 e.agent_id,
 e.has_error,
 e.is_rework,
 r.resource_name,
 r.department
 FROM event e
 LEFT JOIN resource r ON e.resource_id = r.resource_id
 WHERE e.process_id = %(process_id)s
 """
 params = {"process_id": process_id}
 
 if start_date:
 query += " AND e.event_timestamp >= %(start_date)s"
 params["start_date"] = start_date
 if end_date:
 query += " AND e.event_timestamp <= %(end_date)s"
 params["end_date"] = end_date
 
 query += " ORDER BY e.case_id, e.event_timestamp"
 
 df = pd.read_sql(query, engine, params=params)
 
 df = pm4py.format_dataframe(
 df,
 case_id='case_id',
 activity_key='activity_name',
 timestamp_key='event_timestamp'
 )
 
 return df


def compute_automation_kpi(df, measurement_date, process_id):
 """DataFrameからautomation_kpiテーブル用のKPIを算出"""
 
 results = []
 
 for activity in df['activity_name'].unique():
 act_df = df[df['activity_name'] == activity]
 
 total = len(act_df)
 automated = len(act_df[act_df['resource_type'].isin(['ai_agent', 'rpa_bot'])])
 manual = len(act_df[act_df['resource_type'] == 'human'])
 
 if 'end_timestamp' in act_df.columns:
 act_df_with_dur = act_df.dropna(subset=['end_timestamp']).copy()
 act_df_with_dur['duration_sec'] = (
 act_df_with_dur['end_timestamp'] - act_df_with_dur['event_timestamp']
 ).dt.total_seconds()
 
 auto_dur = act_df_with_dur[
 act_df_with_dur['resource_type'].isin(['ai_agent', 'rpa_bot'])
 ]['duration_sec']
 manual_dur = act_df_with_dur[
 act_df_with_dur['resource_type'] == 'human'
 ]['duration_sec']
 
 results.append({
 'measurement_date': measurement_date,
 'process_id': process_id,
 'activity_name': activity,
 'total_executions': total,
 'automated_executions': automated,
 'automation_rate': automated / total if total > 0 else 0,
 'avg_automated_duration_sec': auto_dur.mean() if len(auto_dur) > 0 else None,
 'avg_manual_duration_sec': manual_dur.mean() if len(manual_dur) > 0 else None,
 'time_saving_rate': (
 1 - (auto_dur.mean() / manual_dur.mean())
 if len(auto_dur) > 0 and len(manual_dur) > 0 and manual_dur.mean() > 0
 else None
 ),
 'automated_error_rate': (
 len(act_df[(act_df['resource_type'].isin(['ai_agent','rpa_bot'])) & (act_df['has_error'])]) / automated
 if automated > 0 else None
 ),
 'manual_error_rate': (
 len(act_df[(act_df['resource_type']=='human') & (act_df['has_error'])]) / manual
 if manual > 0 else None
 ),
 'rework_rate': len(act_df[act_df['is_rework']]) / total if total > 0 else 0,
 })
 
 return pd.DataFrame(results)

6.3 Before/After比較の仕組み

baseline_snapshotテーブルにBefore期間（AI導入前）のKPI値をJSONB形式で固定保存する。ダッシュボード上では、最新のautomation_kpiとbaseline_snapshotの差分を「改善率」として表示する。Before期間のデータは上書きされないよう、スナップショットとして凍結する設計とする。

SELECT 
 k.activity_name,
 (b.kpi_data->>'avg_manual_duration_sec')::NUMERIC AS before_duration,
 k.avg_automated_duration_sec AS after_duration,
 ROUND(
 (1 - k.avg_automated_duration_sec 
 / (b.kpi_data->>'avg_manual_duration_sec')::NUMERIC) * 100, 1
 ) AS improvement_pct
FROM automation_kpi k
JOIN baseline_snapshot b ON k.process_id = b.process_id AND b.is_active = TRUE
WHERE k.measurement_date = (SELECT MAX(measurement_date) FROM automation_kpi)

7.2 Metabase vs Streamlitの使い分け

Metabase（ページ1, 3, 4）はSQLベースのグラフ、KPIカード、フィルタなどの定型ダッシュボードに使用する。PostgreSQLに直結してノーコードでクエリを作成できるため、クライアントへの配信や社内の非エンジニアも使いやすい。Embed機能でクライアントのポータルに埋め込むことも可能。

Streamlit（ページ2, 5）はpm4pyの出力（DFG図、Petri Net、プロセスツリー）をインタラクティブに描画するために使用する。Pythonネイティブなので、pm4pyのvisualizationメソッドをそのまま呼び出してグラフ化できる。Graphviz、Plotly、NetworkXとの連携でプロセスマップを動的に生成する。

8. データ収集設計（Before測定）

8.1 収集方式の優先順位

AI導入前のBeforeデータ収集は、本プロジェクトの成否を左右する最重要フェーズである。

第一優先: 既存システム監査ログの自動取得 — 担当者の手間がゼロであり、データの正確性が最も高い。PostgreSQL/MySQL/SQL Serverのトランザクションログ、Microsoft Graph APIのSharePoint/Dynamics監査ログ、kintoneのレコード更新履歴等から、ステータス変更・レコード更新を抽出してイベントログに変換する。AirflowのDAGで日次自動取得。

第二優先: タスクマイニングによるサンプリング録画 — システムにログが残らない手作業のBefore測定に使用する。OpenRPA（OSS）またはPythonスクリプト（pyautogui + OpenCV）で、代表的な担当者2〜3人に数日間の録画を依頼する。

第三優先: 簡易記録フォーム — 上記で対応できない業務にのみ限定使用。Streamlitで作成した簡易Webフォーム（「開始→終了」ボタン）で記録。データはFastAPI経由でPostgreSQLに直接格納。
